# Recommend papers

Keep research scoping, metadata triage, final comparison, scoring, and synthesis in the controlling Codex. By default, delegate full-paper reading to one external Claude CLI process per paper, matching TASTE Reading; these are Claude processes, never Codex subagents. Only when Claude is unavailable or the user explicitly forbids Claude, use exactly three Codex subagents in the fast batch fallback described below. Treat an explicit user choice to disable Claude as sticky for the entire current Codex conversation, including later questions, new research directions, same-run continuations, and child runs. Do not return to Claude merely because a later turn is silent; change back only when the user explicitly re-enables Claude, unless the original instruction explicitly limited itself to one turn. Use the bundled backend for crawling, normalization, identity validation, caching, artifact validation, and run state.

## Runtime

The backend must work on Windows, macOS, and Linux. Do not replace its cross-platform Python entry points or locks with platform-specific shell commands or operating-system APIs.

Resolve `SERVICE` by taking the absolute source locator of this `SKILL.md` and appending `scripts/paper_service.py`. Resolve only the service bundled with this installed skill; do not derive it from the current working directory or search the workspace for an alternative implementation.

Resolve one Python command prefix and reuse it for the entire run. Prefer the installer-managed interpreter at the platform data location (`~/Library/Application Support/TASTE/recommend-papers/venv/bin/python` on macOS, `${XDG_DATA_HOME:-~/.local/share}/taste/recommend-papers/venv/bin/python` on Linux, or `%LOCALAPPDATA%\TASTE\recommend-papers\venv\Scripts\python.exe` on Windows). Otherwise consider: (1) an absolute `RECOMMEND_PAPERS_PYTHON` executable supplied by the user; (2) the active Python 3.10+ environment; (3) `<SKILL_DIR>/../../.venv/bin/python` or its Windows counterpart when present; (4) standard versioned Python commands; (5) an available Conda environment. A prefix may be one executable path or a launcher plus version arguments such as `py -3.12`. For each candidate, run `paper_service.py doctor`:

- Select it immediately when `status` is `ok`.
- When it returns `error_type: runtime_setup` with `missing_module`, select that Python 3.10+ candidate. Install the reported `requirements` with the exact same command prefix only when it belongs to a user-writable isolated environment; otherwise create an isolated venv first. Then rerun `doctor`.
- Reject candidates that report an unsupported Python version. Stop and report non-setup failures such as unwritable storage instead of hiding them by changing Python.

The environment manager and environment name are not requirements. Never switch Python command prefixes midway through a run. Use native invocation syntax: `"$PYTHON" "$SERVICE" ...` for a single executable in POSIX shells, `& $Python $Service ...` in PowerShell, or the equivalent argument-vector execution provided by the current Codex tool. Do not store a launcher and its arguments in one quoted string.

OpenReview always uses the official `openreview-py` client. Prefer authenticated access when both credentials are present in the private `read.env` beside this file; otherwise use the anonymous official client automatically. The user may edit `read.env`, provide `OPENREVIEW_USERNAME` and `OPENREVIEW_PASSWORD` in the process environment, or set `RECOMMEND_PAPERS_OPENREVIEW_ENV_FILE` to an absolute alternative. Never print credentials, copy them into a run, or include them in recommendations.

Keep the skill directory immutable during use. Follow the XDG base-directory split for every runtime write:

- Intermediate runs, resumable staging, receipts, locks, and HTTP state: `${XDG_STATE_HOME:-~/.local/state}/taste/recommend-papers`
- Reusable metadata, full text, and validated single-paper reads: `${XDG_CACHE_HOME:-~/.cache}/taste/recommend-papers`

`RECOMMEND_PAPERS_STATE_DIR` and `RECOMMEND_PAPERS_CACHE_DIR` may override the two application roots with absolute paths. `RECOMMEND_PAPERS_DATA_DIR` remains a backward-compatible alias for the state root only. Never store task runs or reusable article data inside the skill, `CODEX_HOME`, the current working directory, or a Git repository.

The cache root must contain exactly two directories: `metadata` for discovery metadata and `fulltext` for acquired full text plus validated single-paper reading artifacts. Metadata is organized directly by channel: `arxiv/<category>/<day>.json`, `biorxiv/`, `nature/`, `science/`, `journal/`, and `conference/<venue>/<year>.json`. Never create `sources`, `imports`, `finding-runtime`, top-level `http-state`, or hidden state inside `metadata`. Non-cache throttling and migration state belongs under the external data root's `state/`. Run `migrate-metadata-cache` after `doctor`; it canonicalizes complete caches and removes obsolete or duplicate artifacts.

Never write runtime artifacts into TASTE, the current workspace, or another Git repository unless the user explicitly requests an export. The backend rejects any Git-contained write by default.

## Defaults

Apply explicit user settings first, then `references/defaults.json`:

- Defaults are fallbacks, never minimums. Preserve explicit current-turn settings, then established conversation settings, then reusable parent-run scope; fill only still-unspecified fields from `defaults.json`.
- Preserve the conversation's reading-backend lock independently of topic, source, date, count, run, or workflow mode. Interpret “不要用 Claude 精读”, “别用第一种精读”, and equivalent explicit refusals as `reading_preference: codex_fast`, `user_disabled_claude: true`, `reading_preference_scope: conversation`, and `conversation_reading_preference_locked: true`. Keep writing those values into every later plan until the user explicitly asks to use Claude again. User silence never unlocks the preference.
- In `comprehensive` mode, the metadata shortlist defaults to 100 and final recommendations default to 20. Focused and incremental modes choose counts proportionally to the current evidence need.
- In `comprehensive` mode, when the user specifies neither time nor channels, the default scope is all three major conferences—NeurIPS/NIPS, ICLR, and ICML—at each venue's latest single year with usable complete metadata, plus arXiv over the inclusive trailing six calendar months. Focused/incremental modes may choose a narrower justified scope without pretending it is exhaustive.
- Under that default scope, independently add exactly one to three highest-value topic-specific channels and record why. Extra conferences use their latest usable complete-metadata year; extra preprint/journal streams use the same concrete six-month dates as arXiv.
- Conferences: exactly the latest single year with usable metadata; resolve it with `probe-venue`. Always probe the requested/current year through every configured live channel before considering an older year. A future proceedings date or a missing archival directory is never proof that accepted-paper, OpenReview, virtual-conference, OJS, or index metadata is unavailable. Every included conference source must set `complete_catalog: true` and crawl the entire official venue catalog. Never put `limit`, `max_results`, or sampling parameters on a conference source.
- The built-in priority conference catalog is NeurIPS/NIPS, ICLR, ICML, SIGKDD/KDD, SIGIR, CIKM, AAAI, ICCV, WWW, CVPR, ACL, IJCAI, ECCV, and EMNLP. Use each venue's bundled official adapter; never substitute a title-only DBLP corpus. A priority conference cache is usable only when all records have abstracts and the adapter proves full-catalog exhaustion. DBLP-backed title-pool discovery is serial, shares cooldown across all three official mirrors, rotates mirrors on network/5xx failures, and defers an excessive `Retry-After`; do not bypass the shared HTTP client with direct requests.
- When diagnosing a 2026 source, consult `references/live-source-validation-2026-07-22.md` as an observed baseline, then re-run `probe-venue`; the snapshot is evidence of tested routes, not permission to skip a current live probe.
- arXiv: inclusive trailing six calendar months using concrete ISO dates. Unless the user explicitly supplies categories, exhaustively acquire `cs.AI`, `cs.LG`, `stat.ML`, `cs.CL`, `cs.CV`, `cs.IR`, `cs.RO`, `eess.SY`, `cs.MA`, and `cs.NE`. This deliberately covers core AI plus NLP (`cs.CL`), computer vision (`cs.CV`), retrieval/RAG (`cs.IR`), and embodied intelligence through robotics and systems/control (`cs.RO` + `eess.SY`). Never apply task keywords or a result limit during this base crawl. Cache complete daily shards under `metadata/arxiv/<category>/<YYYY-MM-DD>.json`; task-specific title/abstract relevance is decided later by Codex.
- An arXiv day is reusable only after the local calendar day has closed (`shard_date < today`). The current day and any future day must be written `complete: false`, `provisional: true` and refetched on every later crawl; API pagination exhaustion never makes an open calendar day complete. Page checkpoints live under the external data root at `state/arxiv-staging/`, survive failed requests, and are deleted immediately after their month/range is successfully published into daily shards. Never create `metadata/arxiv/.staging`.
- bioRxiv: inclusive trailing six calendar months using concrete ISO dates. Exhaustively cache every category without task-keyword filtering or result limits as `metadata/biorxiv/<YYYY-MM-DD>.json`. The current day is always provisional and refetched; the preceding three closed days are reconciled at most every 24 hours; older proven-complete days are reused indefinitely unless explicitly refreshed. Cursor checkpoints live only under `state/biorxiv-staging/`, survive failed ranges, and are deleted after successful daily-shard publication.
- Download and read every shortlisted paper with available full text. Default Claude mode produces one validated Chinese `papers/<index>/read.md` per paper and aggregates them; the explicit/no-Claude fast mode instead produces three balanced batch-result files and no per-paper reading artifacts.
- Cache: reuse entries no older than seven days by default; user refresh overrides this.

Read `references/migration-parity.md` and `references/service-contract.md` before commands, `references/research-modes.md` for every follow-up, incremental, focused, or partial-stage task, and `references/scoring-rubric.md` before scoring. Never weaken a locked migrated invariant for convenience.

## Multi-turn orchestration

Treat the workflow as composable stages. First inspect the conversation and any known run artifacts. If existing validated evidence answers the new question, reuse it and do not crawl or reread merely to exercise the skill. If the user changes direction, dates, corpus, or ranking objective, create a child run with `init-run --parent-run-dir <prior-run> --mode incremental --question ...`; keep the completed parent immutable and search only the evidence gap. Global caches automatically reuse matching metadata, full text, and default Claude reads.

Write `workflow` into every plan with `mode`, `research_question`, `rationale`, `stop_after`, `reading_preference`, and any per-turn `shortlist_target`/`final_target`. When a conversation-level reading choice exists, also write `user_disabled_claude`, `reading_preference_scope`, and `conversation_reading_preference_locked`; copy them into every child or later run rather than relying only on the immediately preceding plan. Valid modes are `comprehensive`, `focused`, `incremental`, and `metadata_only`; valid stops are `metadata`, `shortlist`, `fulltext`, `reading`, and `recommendation`. Codex may independently customize sources, dates, categories, counts, cache policy, download workers, reading backend, scoring priorities, output language, and deliverable format. Do not ask the user to specify settings that can be inferred safely from the research question and conversation.

Stop when the requested evidence need is satisfied. Use `finish-stage` for a successful metadata, shortlist, fulltext, or reading endpoint; use `finalize` and `complete` only when ranking/recommendations are requested. Clearly label metadata-only conclusions and acquire full text before answering method, experiment, limitation, or other claims unsupported by abstracts.

## Required workflow

1. Choose the workflow mode and inspect reusable evidence first. Run `doctor` and `migrate-metadata-cache` when backend work is needed. Reuse the active run for a same-scope follow-up; otherwise run `init-run`, optionally with `--parent-run-dir`, `--mode`, and `--question`. Require the migration inventory to report `unified: true`.
2. Interpret the request. Infer goals, evidence needs, useful adjacent fields, exclusions, and priority among fit, rigor, novelty, reproducibility, recency, and transferability.
   Always write `request_scope` into the plan with boolean `user_specified_time`, boolean `user_specified_channels`, and concrete `as_of_date`. Do not treat a topic, paper count, or phrase such as “调研论文” as an implicit time/channel choice. If both booleans are false, the backend-enforced default is the three major conferences plus six-month arXiv and one to three topic-adaptive channels; do not ask the user to restate these defaults.
3. Use `catalog` and `probe-venue` to resolve useful channels. Comprehensive mode must consider the four baseline channels and then topic-adaptive channels; focused/incremental modes consider only plausible channels for the current evidence gap. Write `channel_decisions` for every channel actually considered. Once a venue/year is selected, preserve Find semantics: acquire and cache its complete metadata pool before topical filtering. Never put topic filters or limits on a selected venue source.
4. Write the plan only to `<run_dir>/plan.json`. Run `metadata --run-dir ... --plan ...`. Inspect every source receipt. Repair failed or weak coverage before scoring; never interpret source failure as absence of literature.
   Metadata crawls can run for many minutes. If command execution returns a live session/cell ID, it is still running: keep waiting/polling that exact session until its exit code is available. Never describe a yielded session as interrupted, never start a replacement metadata command for the same run, and never continue to scoring without the command's final JSON receipt. For bioRxiv require `status: complete`, `exhausted: true`, `truncated: false`, `closed_days_complete: true`, the expected daily-shard count, and an `exhaustion_proof` beginning with `all_closed_days_complete`. If the interval contains today, require it in `provisional_days`; never add a limit or treat API exhaustion as proof that the open day has closed.
   HTTP 429/503 responses are transient backend events: the service must honor `Retry-After`, wait, and retry internally. Do not announce a source problem while retries or a live metadata process remain. Report a source failure only after the command exits and its receipt proves that all backend retry attempts were exhausted.
   For arXiv require `exhausted: true`, `truncated: false`, `closed_days_complete: true`, the expected category×day shard count, and an `exhaustion_proof` beginning with `all_closed_category_days_complete`. If the requested interval includes the local current day, require that date in `provisional_days` and never accept it as a reusable complete shard. A task query may influence later Codex scoring but must not narrow these cached base shards.
5. If the chosen stop is after metadata, call `finish-stage --stage metadata` and answer at metadata evidence level. Otherwise score every paper in `<run_dir>/metadata.json` yourself using the metadata rubric. Use only title and abstract at this stage; do not let prestige, date, identifiers, code links, or PDF availability add ranking points. Run `shortlist --target N` with the plan's target; use 100 only when it remains unspecified in comprehensive mode. If stopping here, call `finish-stage --stage shortlist`.
6. When full-text evidence is needed, run `fulltext --run-dir ... --workers ...`. The worker count is the global download ceiling; unrelated channels run concurrently with independent slots and cooldowns. Same-channel capacity is source-specific and overridable, except OpenReview defaults to one shared slot and is hard-capped at three. Its official-client login, metadata, attachment, direct PDF, and HTML routes share that same cross-process gate and persisted cooldown. The backend honors `Retry-After`; after the initial parallel pass it waits for bounded active cooldowns and requeues affected papers exactly once with one recovery worker. Treat `temporarily_unavailable`/`retryable_after_cooldown` as a transient acquisition result, never as proof that no PDF exists. The backend validates identity/body content, caches successful full text, and resumes by identity. If the requested stop is fulltext, call `finish-stage --stage fulltext`.
7. Select the reading mode before dispatch by scanning the whole current conversation, not only the latest turn. Apply an active conversation-level no-Claude lock before defaults or Claude availability checks. Use external Claude only when no such lock exists. `doctor.claude.available=false` or `claude-reads.status=claude_unavailable` is proof of unavailability; ordinary Claude failures are not permission to switch modes silently. Never launch `claude-reads` anywhere later in the same conversation while the user-disabled lock remains active.
   **Default TASTE/Claude mode:** Run `prepare-reads --run-dir ...`, then `claude-reads --run-dir ... --workers 16`. Every pending paper still receives its own fresh external `claude` CLI process, but at most 16 processes run simultaneously. Keep the 16-slot pool full: whenever one Claude finishes, immediately launch the next queued paper until all are processed. Never create all paper processes at once, never exceed 16 even if a larger `--workers` value is supplied, and do not use Codex `spawn_agent` for this mode. Each Claude receives one deterministic `read_prompt.md`, has `Agent`/`Task` disabled, reads one complete `full_text_path`, and writes only that paper's `read.md` and `read_receipt.json` scientific artifacts. Cached reads are reused only under the current contract and exact full-text SHA-256.
   **Fast three-Codex fallback:** Use this only when Claude is proven unavailable or the user explicitly disabled it. Run `prepare-fast-read-batches --run-dir ...`; it balances all ready papers across exactly three manifests with sizes differing by at most one. Start exactly three Codex subagents—one per manifest—and directly give each the original research request, the scoring priorities, and its manifest; instruct it to read every `full_text_path` in that manifest as quickly as practical, compare its papers, and write the manifest's `result_path` as a JSON array of evidence cards. Do not call `prepare-reads`, `claude-reads`, or `validate-reads`; do not create per-paper `read.md`, per-paper receipts, translation maps, repair agents, or one-paper subagent tasks. Each returned card must include its assigned `batch_id`, identity, title, exact `full_text_path`, concise summary, decisive evidence, limitations, borrowable elements, confidence, 0–10 `match_score`, 0–10 `transferability_score`, and their sum as `final_score`. Wait for exactly those three subagents, merge their arrays into `evidence_cards.jsonl`, and let the controlling Codex normalize scores across batches and resolve the final cutoff. This branch deliberately trades per-paper audit depth for speed.
8. In default Claude mode, every single-paper `read.md` must be Chinese and use exactly: the paper title; source and URL/PDF lines; `摘要`; `动机与核心创新` with `动机：` and `核心创新：`; `方法` with a paper-specific relevant LaTeX formula and plain-language explanation; `实验结果`; and `优缺点总结`. When `abstract_zh` is absent, `摘要` must be the complete Chinese translation of the supplied original English abstract—not a summary—while preserving model names, dataset names, numbers, conclusions, and limitations. The receipt maps every source sentence to its translation. Follow the original TASTE Reading length and Markdown rules in the contract.
9. In default Claude mode, run `validate-reads --run-dir ...`. It applies Claude completion receipts, exact paths, whole-text `evidence_chars`, translation, section, length, formula, hash, and cache gates. For fresh failures, run `claude-reads --run-dir ... --only-failed --workers 16` once: it gives each failed paper one new external Claude with the exact errors, using the same hard-capped refilling pool, and requires minimal repair. Validate again; if failure remains, report it rather than switching to the Codex fallback without authorization.
10. If the requested stop is reading, call `finish-stage --stage reading` after default Claude validation. For recommendation tasks, persist one JSONL evidence card per acquired paper. Default Claude mode cites both exact `full_text_path` and validated `read_path`; fast fallback cites exact `full_text_path` and correct `batch_id`. Run `finalize --target N` with the current-turn target; use 20 only when still unspecified in comprehensive mode.
11. Re-read close calls around the cutoff and correct artifacts when necessary.
12. Write `<run_dir>/recommendations.md` as the final synthesis: cross-paper themes, agreements/conflicts, reusable methods, evidence gaps, and each final recommendation. Run `complete --recommendations ...`.
13. Report the synthesis to the user, including the selected reading mode, coverage, full-text gaps, validated single-paper-read count in Claude mode or three-batch coverage in fast mode, read/scored count, and recommendation shortfall.

## Evidence rules

- Metadata relevance is triage, not scientific quality.
- Preserve paper identity across every fallback; never substitute a similar paper. When current-year metadata predates its official PDF, full-text acquisition may use Semantic Scholar, HAL, or arXiv only on an exact normalized-title match, followed by the same PDF identity/body validation.
- Do not infer full-text results from abstracts.
- Separate reported evidence from Codex inference.
- Record retraction/supersession, identity uncertainty, unsupported claims, or critical missing evidence explicitly in evidence and confidence; do not silently invent extra ranking dimensions beyond the original match/transferability scores.
- Never claim completion while `run.json` is not `status: complete`.
- Never upload or push unless separately requested.
